# PR #98 follow-ups (E2E test 2026-05-07)

PR #98 (`feat/skilify`) wires skilify mining into OpenClaw's `agent_end`
hook (commit 7900eb7). Live E2E against a real Telegram-driven OpenClaw
gateway exposed one shipping bug + two latent issues. Fixes for the
shipping bug landed in this branch; the two latent issues are tracked
here for follow-up PRs.

## Fixed in this PR (commits 7900eb7 + this branch's tail)

### Bug — gate-runner had no dispatch for `agent: "openclaw"`

**Symptom:** worker spawned, fetched sessions, extracted Q&A pairs,
then died at the gate:

```
running gate (agent=openclaw, bin=null, prompt=48580 chars)
gate failed: agent binary not found at undefined (agent=openclaw) (stdout=0, stderr=0)
```

**Why CI + the contributor's E2E missed it:** the contributor's test
(`claude-code/tests/skilify-session-start-injection.test.ts`) is a
bundle scan — it asserts `dist/index.js` contains the right markers
but never actually invokes the worker. Their hand-driven E2E ran the
worker directly with `agent: "claude_code"` substituted in (per the
commit message: *"agent_label 'claude_code' since pi/openclaw aren't
installed in the test env"*), so the openclaw → spawn → worker → gate
chain was never exercised end-to-end before this run.

**Fix:** split worker config into `agent` (provenance label, can be
"openclaw") + optional `gateAgent` (CLI dispatch, must be one of the
five real CLI agents — claude_code/codex/cursor/hermes/pi). Worker
passes `cfg.gateAgent ?? cfg.agent` to `runGate`; provenance fields
(`sourceAgent` in skills table, `created_by_agent` in SKILL.md
frontmatter) keep using `cfg.agent`. OpenClaw spawn helper detects
the delegate via `which claude || which codex || ...` at spawn time
and threads it through. Other agents (CC/Codex/Cursor/Hermes/Pi) need
no change — `gateAgent` is undefined → falls back to `agent`.

**Bails out cleanly when no delegate is available:** if the user runs
OpenClaw on a machine with none of the five CLIs on PATH, the spawn
helper logs a warning and skips spawning. Better than firing a worker
that's guaranteed to die at the gate.

**Live verification (real Telegram message, real Deeplake org):**

**Regression guard:** `claude-code/tests/skilify-session-start-injection.test.ts`
gained a new test asserting (a) `detectOpenclawGateAgent` exists and lists
all five candidate CLIs, (b) the spawn helper bails with a clear message
when none is found, (c) `gateAgent` is threaded into the worker config,
(d) the bundled `openclaw/dist/index.js` preserves all of the above.

### Discoverability — `hivemind_setup` output now mentions skilify

OpenClaw is structurally different from the other six agents: it has no
session-start prompt-injection mechanism and no Bash tool. So the
banner-text-and-Bash discovery path the other agents use (which lists
`hivemind skilify status / pull / scope / ...` as CLI commands the agent
shells out to) is unreachable from inside an OpenClaw chat. Without a
hint somewhere, OpenClaw users wouldn't know mining was happening or
that they could pull teammates' skills.

Solution: extend the existing `/hivemind_setup` slash command's success
text with a short pointer to the CLI:

> Skill mining (skilify) runs in the background after each turn — your
> conversations get crystallised into reusable skills automatically.
> From your terminal:
>   hivemind skilify status   — see what's been mined
>   hivemind skilify pull     — fetch teammates' skills

This honours the design principle established by the contributor's
comment ("centralize commands to CLI bash, avoid the file md") — no new
slash commands or registered tools, just discoverability for the
existing CLI surface.

## Loose ends NOT fixed in this PR

### 1. OpenClaw spawn lock has no staleness check

`openclaw/src/index.ts:tryAcquireOpenclawSkilifyLock` uses
`O_CREAT|O_EXCL|O_WRONLY` — it succeeds only if the lock file does not
exist, and creates an empty file on success. There is no staleness
check.

If a worker dies abnormally (host kill, OOM, segfault, anything that
prevents the worker's `releaseWorkerLock` finally-block from running),
the empty lock file persists indefinitely and EVERY subsequent
`agent_end` will silently skip the spawn. Mining stops permanently
for that project_key until manual cleanup.

This is what we hit during this E2E: the first gate-failed run left
the lock behind, and the second test was about to silently skip until
I `rm -f`'d the lock manually.

**Suggested fix (separate PR):** mirror the staleness logic from
`src/skilify/state.ts:tryAcquireWorkerLock` (lines 200-221). If the
lock file exists, read its contents as a millisecond timestamp; if
`Date.now() - parsed > maxAgeMs` (or unparseable → treat as stale),
unlink and recreate. The OpenClaw spawn helper currently writes an
empty file, so all existing locks would be treated as immediately
stale, which is the right behavior. Need to also start writing a real
timestamp when the spawn helper creates a fresh lock.

### 2. The `Auto-update check failed: timeout` regression flagged in PR #97 follow-ups still reproduces

Pre-existing, NOT introduced by this PR. Documented separately in
`.followups-pr97.md`. Confirmed still reproduces against the new
bundle (07:32:32 and 07:42:13 of this E2E run). Same root causes
remain plausible — cold TLS + bonjour-watchdog noise concurrent with
the version check on a 5-second AbortSignal timeout.

## Confidence

**95%** that the gate-agent delegation works correctly for the common
case (OpenClaw user has Claude Code installed alongside). Backed by
live E2E producing a real verdict, real SKILL.md, real Deeplake row
with correct provenance.

**Untested scenarios:**

- OpenClaw user with ONLY codex / cursor-agent / hermes / pi installed
  (no claude). The detection function tries them in order so this
  *should* work, but only the claude_code branch was exercised live.
- OpenClaw user with NONE of the five CLIs. The spawn helper has the
  bail-out branch but it wasn't traversed in this E2E.
- The lock-staleness latent issue (loose end #1). Will manifest after
  the first abnormal worker death on a machine that's never seen one;
  by definition we couldn't hit it cleanly in a single E2E run.
- Concurrent agent_end events on different projects (multi-channel
  OpenClaw). Each project_key has its own lock so they should be
  independent, but only one channel was driven in this run.
